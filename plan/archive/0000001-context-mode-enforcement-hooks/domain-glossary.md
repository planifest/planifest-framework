## Terms

| Term | Definition | Aliases | Used In |
|------|-----------|---------|---------|
| PreToolUse hook | A Claude Code extension point that fires synchronously before a tool call is executed. The hook script reads tool invocation data from stdin and writes a decision to stdout. | hook, tool hook | REQ-001, REQ-002, REQ-003, REQ-004, settings.json |
| block decision | A hook response with `hookSpecificOutput.permissionDecision: "deny"` that prevents the tool call from executing and surfaces `permissionDecisionReason` to the agent. Exit 0 required — exit 2 is a hard error, not a block signal. | hard block, deny, enforcement | REQ-001, REQ-002, REQ-003, all hook scripts |
| allow decision | Exit 0 with no stdout output, or `hookSpecificOutput.permissionDecision: "allow"`. Permits the tool call to proceed. Returned by REQ-002 when the Bash command matches no blocked patterns. | pass-through | REQ-002 block-bash.sh |
| enforcement hook | A PreToolUse hook whose purpose is to block disallowed tool usage and redirect to the correct sandbox alternative. Distinct from advisory hooks that warn but do not block. | blocking hook | all hook scripts |
| routing rules | The agent instruction file (AGENTS.md or equivalent) installed by `--context-mode-mcp` that tells agents which `ctx_*` tools to use. Advisory only — not enforced by code. The enforcement hooks are the hard enforcement layer. | context-mode agents file | setup.sh, AGENTS.md |
| allowlist | The set of Bash command prefixes (`git`, `mkdir`, `rm`, `mv`, `cd`, `ls`, `npm install`, `pip install`) that are never blocked by the Bash hook, regardless of other content in the command string. | safe commands | REQ-002, block-bash.sh |
| blocked pattern | A command substring or token that causes the Bash hook to return a block decision: `grep`, `rg`, `curl`, `wget`, `http://`, `https://`. | forbidden pattern | REQ-002, block-bash.sh |
| context-mode MCP | The upstream MCP plugin at `mksglu/context-mode` that provides sandbox tool execution (`ctx_execute`, `ctx_execute_file`, `ctx_batch_execute`, `ctx_fetch_and_index`, `ctx_search`). Assumed configured by the user independently of setup. | context-mode | feature-brief.md, design.md |
| ctx_execute | The context-mode MCP tool for running shell commands or code in a sandboxed environment. Replaces `Bash` for large-output commands and `Grep` for codebase searches. | sandbox execute | REQ-001, REQ-002, block-grep.sh, block-bash.sh |
| ctx_fetch_and_index | The context-mode MCP tool for fetching a URL and indexing its content into the knowledge base. Replaces `WebFetch`. | sandbox fetch | REQ-003, block-webfetch.sh |
| ctx_search | The context-mode MCP tool for querying the indexed knowledge base. Used after `ctx_fetch_and_index` to retrieve content. | sandbox search | REQ-003, block-webfetch.sh |
| hook script | A bash executable that implements a PreToolUse hook. Reads JSON from stdin, writes a JSON decision to stdout, exits 0. | enforcement script | block-grep.sh, block-bash.sh, block-webfetch.sh |
| hook wiring | The `PreToolUse` entries in `.claude/settings.json` that associate a tool name with a hook script path. Written by setup integration when `--context-mode-mcp` is passed. | settings wiring, hook config | REQ-004, setup.sh, setup.ps1 |
| context-mode-hooks (component) | The planifest-framework component consisting of the three hook bash scripts. Source location: `planifest-framework/hooks/context-mode/`. Installed location: `.claude/hooks/context-mode/`. | hook scripts component | component manifest |
| setup-hook-integration (component) | The planifest-framework component consisting of changes to `setup.sh`, `setup.ps1`, and `setup/claude-code.*` that install hook scripts and write hook wiring. | setup integration component | component manifest |
