# ADR-005 - Exit-0 failure mode for all hook scripts

## Context

Hook scripts run in the critical path of every agent tool call. If a hook script crashes, throws an unhandled exception, or exits non-zero for an unexpected reason, the agentic tool may surface an error to the human, refuse to continue, or enter a degraded state. This is a worse outcome than the hook silently failing.

There is a tension between:
- **Observability:** surfacing hook errors so developers can diagnose and fix them
- **Continuity:** never letting hook failures interrupt a working session

The telemetry hooks (`emit-phase-start.mjs`, `emit-phase-end.mjs`) are entirely optional — their failure should never be visible. The enforcement hooks (`gate-write.mjs`, `check-design.mjs`) use non-zero exit codes intentionally (exit code 2 for blocks) but must still exit 0 for all unexpected failure paths.

---

## Decision

All Planifest hook scripts follow this failure policy:

1. **Expected blocks** (enforcement hooks only): exit code 2 with a human-readable message. This is intentional behaviour, not a failure.
2. **All other errors** (network timeout, file read failure, JSON parse error, unexpected exception, missing env vars, read-only tmpdir): catch at the top level, exit 0 silently. No output to stdout or stderr.
3. **3-second HTTP timeout** (telemetry scripts): AbortController fires → silent exit 0.
4. **Missing sentinel** (`PLANIFEST_TELEMETRY_URL` unset): early return, exit 0.

The only non-zero exit codes any hook script produces are the intentional exit-2 blocks in `gate-write.mjs`.

---

## Alternatives Considered

| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|-------------|
| Exit non-zero on all errors (fail loudly) | Full observability; errors are never silent | Any script bug or environmental issue blocks agent sessions; unacceptable regression | Session continuity is non-negotiable |
| Exit non-zero for enforcement hooks; exit 0 for telemetry | Consistent within each category | Enforcement hook bugs could block sessions; hard to distinguish intentional blocks from bugs | Risk of accidental blocking from non-enforcement errors |
| Write errors to a log file | Preserves observability without blocking | Adds filesystem dependency; log file location varies by platform; adds complexity | Out of scope for v1; can be added as an enhancement |
| Emit a `hook_error` telemetry event on failure | Structured observability | Requires telemetry to be working (circular dependency if telemetry hook itself fails); adds complexity | Deferred; telemetry hooks must exit 0 even if telemetry is down |

## Affected Components

| Component | Impact |
|-----------|--------|
| `planifest-framework/hooks/telemetry/emit-phase-start.mjs` | Top-level try/catch; all errors exit 0 |
| `planifest-framework/hooks/telemetry/emit-phase-end.mjs` | Top-level try/catch; all errors exit 0 |
| `planifest-framework/hooks/enforcement/gate-write.mjs` | Top-level try/catch; only exit 2 for intentional blocks |
| `planifest-framework/hooks/enforcement/check-design.mjs` | Top-level try/catch; always exit 0 |
| All adapter scripts | Must propagate exit codes correctly: pass exit 2 through; catch all other non-zero exits and convert to 0 |

## Consequences

**Positive:**
- Agent sessions are never blocked by hook infrastructure failures
- Enforcement hooks can be deployed to existing projects without risk of disruption
- CI environments without telemetry configured are silently unaffected

**Negative:**
- Silent errors make debugging difficult; a misconfigured hook may appear to work (returns 0) while doing nothing
- Enforcement hooks failing silently means the gate is effectively disabled during the failure period — scope drift is undetected

**Risks:**
- An adapter that incorrectly converts exit-2 blocks to exit-0 silently disables enforcement. This is the most dangerous failure mode of the policy (see ADR-001, adapter correctness risk).
