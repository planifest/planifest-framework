# ADR-006 - Two-registry model for capability skills

**Tool:** Claude Code
**Model:** claude-sonnet-4-6
**Feature:** 0000005-framework-governance
**Status:** accepted
**Date:** 02 May 2026

---

## Context

Humans need to introduce external capability skills mid-pipeline — for example, a specialised frontend design skill or a domain-specific code generator. Some skills are useful for one plan only; others should be available permanently. The intake mechanism (skills-inbox) processes them, but a decision is needed on how skills are stored, discovered, and scoped.

---

## Decision

Two registries: `plan/current/external-skills.json` (plan-scoped, written by orchestrator when a one-time skill is classified) and `planifest-framework/external-skills.json` (permanent, generated by setup scanning `planifest-overrides/capability-skills/`). Agents check plan-scoped first, permanent second. The orchestrator merges both at each phase transition and surfaces only skills relevant to the upcoming phase and stack. Plan-scoped skills are archived by ship-agent at P7.

---

## Alternatives Considered

| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|-------------|
| Single global registry for all skills | Simpler — one place to look | Plan-specific skills pollute the permanent registry; no clean way to scope a skill to one run | Scoping is a core requirement — one-time skills must not persist across runs |
| No registry — agents scan capability-skills directories directly | No registry file to maintain | Directory scans are slower and noisier than a pre-built index; agents cannot filter by relevance without reading all SKILL.md files | Registry allows setup to pre-index and orchestrator to filter by phase/stack without loading every skill |
| Embed skills directly in design.md | Already read by all agents | design.md is not the right format for a structured skill inventory; embedding SKILL.md content bloats the design document | `## Active Skills` section in design.md references skills by name; full content lives in the registry and skill files |

---

## Affected Components

| Component | Impact |
|-----------|--------|
| planifest-orchestrator | Merges both registries at phase transition; surfaces relevant skills; writes plan-scoped registry |
| All phase agents | Read `## Active Skills` from design.md; load skill from registry on reference |
| planifest-ship-agent | Archives `plan/current/capability-skills/` and `plan/current/external-skills.json` at P7 |
| setup.ps1/setup.sh | Scans `planifest-overrides/capability-skills/`; generates `planifest-framework/external-skills.json` |

---

## Consequences

**Positive:**
- Clean separation between per-run and permanent skills — no cross-contamination
- Plan-scoped skills are archived with the run that used them — reproducibility preserved
- Permanent skills survive pipeline runs without re-introduction

**Negative:**
- Two registries to keep consistent — if setup is not re-run after adding a permanent skill, the registry is stale
- Orchestrator must merge and filter on every phase transition — minor complexity

**Risks:**
- Setup sync gap (R-004 in risk register): human adds a permanent skill to `planifest-overrides/capability-skills/` but forgets to re-run setup — permanent registry is stale until next setup run
